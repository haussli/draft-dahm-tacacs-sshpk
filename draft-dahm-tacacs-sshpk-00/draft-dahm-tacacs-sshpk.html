SSH Public Key Distribution for Device Administration using Terminal Access Controller Access-Control System Plus (TACACS+)

Abstract

SSH [RFC4251] provides robust and reliable mechanism to connect to Network Devices for administration. Conventionally, the Public Keys required to authenticate SSH sessions are provisioned directly on the Network Devices. This document adds an extension into Terminal Access Controller Access-Control System Plus (TACACS+) to replace the need for the direct provisioning of SSH public keys onto the Network Devices.¶

1. Introduction

SSH provides robust and reliable mechanism to connect to Network Devices for administration. The widespread approach to provisioning the Public Keys required to authenticate SSH sessions is to upload them through some administration interface onto the Network Devices which are acting as SSH Server. Once an SSH connection has been established, the Device Administration session is then often mediated (authorized and audited) through protocols such as Terminal Access Controller Access-Control System Plus (TACACS+) Protocol via one or more centralised Servers.¶

This document describes an enhancement of the secured version of the TACACS+ protocol [RFC9887] to facilitate the download of the applicable Public Keys from a central TACACS+ server so that the TACACS+ Client can use it to complete SSH authentication. The intent is to shift the burden of administration of SSH Public Keys from the Network Devices to the central TACACS+ Server. Note that this model does NOT extend the SSH authentication flow out from the SSH Server (The TACACS+ Client), to the TACACS+ Server. The authentication remains completely in the SSH Server (The TACACS+ Client). It is important to keep this in mind, as the phase of the AAA protocol that is used to download the public Keys is actually the TACACS+ Authentication phase.¶

Specifically, this document adds a new authentication action value into TACACS+ Authnentication flow, the reply to which, allows the TACACS+ Server to send the relevant SSH Public Keys to the device.¶

The terms defined in Section 2 of [RFC9887] are fully applicable here and will not be repeated. The following terms are also used in this document.¶

2.1. SSHST+

An SSHST+ (SSH Server/TACACS+ Client). This is the device that is running the SSH Server and is configured as a TACACS+ Client.¶

3. SSH Public Key Distribution

The following mechanism describes the process. This enhancement of TACACS+ Protocol is only applicable to the TLS TACACS+ protocol.¶

3.1. Authentication START Packet

A new enumerated value is added for use within the action field of the Authentication START Packet:¶

TAC_PLUS_AUTHEN_SEND_PUBLIC_KEYS := 0x05
¶

3.2. Authentication REPLY Packet

A New enumerated value is added for use within the status section of the Authentication REPLY Packet:¶

TAC_PLUS_AUTHEN_STATUS_MORE := 0x22
¶

3.3. SSH Key download process

As a result of SSH connection attempt, an SSHST+ requires the public key for a user.¶

If no key is available, then it MAY initiate a TACACS+ Authentication flow to retrieve the key from the TACACS+ Server.¶

SSHST+ sends an Authentication Start packet with the action field set to TAC_PLUS_AUTHEN_SEND_PUBLIC_KEYS defined in Section 3.1:¶

  1. user = username¶

  2. action = TAC_PLUS_AUTHEN_SEND_PUBLIC_KEYS¶

  3. priv_lvl = 0 XXX This field might not be 0; right?¶

  4. authen_type = TAC_PLUS_AUTHEN_TYPE_ASCII¶

  5. service = TAC_PLUS_AUTHEN_SVC_LOGIN¶

The user field of the Authentication Start packet contains the username for which the public keys are required.¶

The TACACS+ Server processes the start packet, extracting the username, and determines if it can provide the public keys which are applicable for the username. The source of public keys is not defined by this document.¶

If public keys are not available, the server responds with status=TAC_PLUS_AUTHEN_STATUS_FAIL.¶

If public keys are available, the server responds with status=TAC_PLUS_AUTHEN_STATUS_PASS if there is only one applicable key or status=TAC_PLUS_AUTHEN_STATUS_MORE if there are more than one key. The Public Keys are encoded into the data field as described in Section 3.4.¶

  1. status = TAC_PLUS_AUTHEN_STATUS_PASS¶

  2. server_msg = EMPTY¶

  3. data (see Section 3.4)¶

XXX We had a proposal for this; "PASS" seems weird.¶

SSHST+ receives the Authentication Reply packet and extracts the public key from the data field.¶

SSHST+ uses the public key to authenticate the user. If there is a failure, but the status is TAC_PLUS_AUTHEN_STATUS_MORE, then the SSHST+ can send an Authentication CONTINUE Packet with the data field set to the last SSH Key that was returned. The cycle can continue until the authentication passes or the Server runs out of keys. When the TACACS+ Server sends the last key (or first key if there is only one), it MUST respond with TAC_PLUS_AUTHEN_STATUS_PASS.¶

Authorisation and Accounting proceed as they would if the SSH key had been provisioned by the administrator directly onto the SSHST+.¶

3.4. Encoding of SSH Public Key in Authentication Reply Packet

The Public Key is encoded in the data field of the Authentication Reply Packet as follows:¶

  1. The encoding is as a Text String and must conform to Section 3.7 of [RFC8907].¶

  2. The content is limited to 32716 characters. Any larger, the key MUST be treated as unavailable as described in Section 3.3.¶
